## üîÑ Sincronizaci√≥n Incremental

### ¬øC√≥mo funciona?

La librer√≠a utiliza una **estrategia de sincronizaci√≥n incremental inteligente** que evita descargar miles de productos innecesariamente:

1. **üìÖ Obtiene la fecha de √∫ltima sincronizaci√≥n** desde la base de datos
2. **üîç Crea un filtro Odoo** para productos modificados: `[['write_date', '>', √∫ltima_fecha]]`
3. **üì• Descarga solo productos modificados** despu√©s de esa fecha
4. **üîÑ Aplica estrategia UPSERT** para manejar productos nuevos vs existentes

### Escenarios de Sincronizaci√≥n

| Escenario | Productos Descargados | Frecuencia | Eficiencia |
|-----------|----------------------|------------|------------|
| üöÄ **Primera sincronizaci√≥n** | 10,000 (todos) | Una vez | 0% (necesario) |
| üìÖ **Sincronizaci√≥n rutinaria** | 15-50 productos | 99% de las veces | 99.5% |
| üìà **D√≠a con muchos cambios** | 150-500 productos | Ocasional | 95% |
| üò¥ **Sin cambios** | 0 productos | Frecuente | 100% |
| üîß **Forzar completa** | 10,000 (todos) | Solo manual | 0% |

### C√≥digo de Ejemplo

```python

# Solo descarga productos modificados desde la √∫ltima sincronizaci√≥n
```

## üîÑ Manejo de Productos: Nuevos vs Existentes

### Estrategia UPSERT

La librer√≠a **no diferencia manualmente** entre productos nuevos y existentes. En su lugar, utiliza la estrategia **UPSERT** de PostgreSQL:

```sql
INSERT INTO products (sku, name, description, ...)
VALUES (?, ?, ?, ...)
ON CONFLICT (sku) DO UPDATE SET
    name = EXCLUDED.name,
    description = EXCLUDED.description,
    list_price = EXCLUDED.list_price,
    last_update = EXCLUDED.last_update;
```

### Comportamiento con SKUs Duplicados

| Situaci√≥n | Comportamiento | Resultado |
|-----------|---------------|-----------|
| üÜï **SKU nuevo** | `INSERT` | Producto creado |
| üîÑ **SKU existente** | `UPDATE` | Producto actualizado |
| üì¶ **Lote mixto** | `UPSERT` | Algunos INSERT, algunos UPDATE |
| ‚ùå **Sin cambios** | `UPDATE` | Sin modificaciones reales |

### Ventajas de esta estrategia:

‚úÖ **Simplicidad**: No necesita l√≥gica compleja de comparaci√≥n
‚úÖ **Eficiencia**: PostgreSQL optimiza autom√°ticamente las operaciones
‚úÖ **Robustez**: Maneja todos los casos edge autom√°ticamente
‚úÖ **Escalabilidad**: Funciona igual con 10 o 10,000 productos
